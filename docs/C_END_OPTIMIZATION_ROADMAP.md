结论很直接：Koto 当前不缺“单点能力”，缺的是能让普通用户持续依赖的任务闭环。

### 新的优先级判断

此前路线图里大量功能偏“文件管理产品增强”，但以当前代码现实看，Koto 更有机会先成为：

1. 真正替用户持续跟进事情的个人助手
2. 有授权边界的跨应用行动代理
3. 低打扰但高命中的主动协助系统

因此，后续 C 端优先级从高到低调整为：

1. 持续代办执行闭环
2. 权限化跨应用行动
3. 个人世界模型
4. 屏幕与上下文理解
5. 高命中主动协助
6. 信任与可回放层

## 🎯 现在最该补的 6 个能力

### P0. 持续代办执行闭环

目标不是“回答一个请求”，而是“接下一件事并持续盯到结束”。

典型场景：
- 盯价格、盯快递、盯签证、盯招聘、盯报销、盯订阅续费
- 用户说一次，Koto 跨天继续推进
- 必要时回来只问关键确认，不重复全量上下文

最小能力定义：
- 创建长期任务
- 自动拆分阶段
- 记录状态与截止时间
- 定时检查/触发执行
- 在关键节点回到用户面前
- 输出最终结果、证据、下一步建议

这是最适合优先落地的能力，因为它能直接把现有 Agent、记忆、触发器串起来。

### P0. 权限化跨应用行动

Koto 要从“会分析”变成“会代办”，必须支持授权后的外部行动：
- 浏览器打开与标签整理
- 日历/提醒事项创建
- 邮件/消息草稿生成
- 本地文件归档、导出、重命名、搬运
- 表单填写与步骤引导

关键不只是动作本身，而是：
- 每个动作都有风险分级
- 高风险动作先预览再确认
- 所有动作有结果记录
- 支持失败恢复和重试

### P1. 个人世界模型

当前记忆更偏“画像和偏好”，下一步要补成“生活与工作结构”。

应该被结构化记住的内容：
- 常用联系人
- 常用应用/网站
- 固定文件目录与项目目录
- 常见支付/订阅项目
- 工作时间、休息时间、提醒容忍度
- 周期性目标与长期事项

有了这层，Koto 才能在主动协助时真正命中，而不是泛泛提醒。

### P1. 屏幕上下文理解

普通用户大量需求都来自“眼前这一屏”。

应支持：
- 当前活跃窗口识别
- 截图/OCR 理解
- 网页主体内容提取
- 剪贴板上下文吸收
- 报错弹窗/表单状态判断

有这一层，Koto 才能从“等用户解释”变成“看懂你卡在哪”。

### P1. 高命中主动协助

当前触发器系统已经是不错的基础，但产品上还要再补三件事：
- 是否值得打扰的决策模型
- 主动内容模板化与场景化
- 根据用户反馈持续降噪

主动协助应该少而准，例如：
- 你明早有会，我已整理完相关材料
- 这个月有一个订阅即将自动续费
- 你今天连续做了三次同类处理，要不要我帮你批量化

### P1. 信任与回放层

这层不是“锦上添花”，而是 C 端是否愿意授权的前提。

必须具备：
- 动作预览
- 风险等级
- 二次确认
- 审计日志
- 结果回放
- 撤销/补救建议

## 🚀 三周最小可落地版本

### 目标

三周内做出一个用户能真实感知的能力：

> 用户交代一件持续性任务，Koto 能创建任务、自动跟进、在关键节点主动回来、并给出结果摘要。

### Week 1: 长任务对象模型

新增内容：
- `Task` / `TaskRun` / `TaskCheckpoint` 数据结构
- 任务状态机：`draft` / `active` / `waiting_user` / `completed` / `failed`
- 任务创建 API
- 任务列表与详情 API
- 基础持久化（JSON 或 SQLite，优先沿用当前轻量方案）

验收标准：
- 用户能通过一句自然语言创建长期任务
- 每个任务有状态、截止时间、最近动作、下次检查时间

### Week 2: Agent + Trigger 串联

新增内容：
- 定时扫描活跃任务
- 到检查点后调用 `UnifiedAgent` 执行一次跟进
- 将结果写入任务日志
- 遇到缺信息时把任务置为 `waiting_user`

验收标准：
- 至少支持 3 类任务模板：价格监控、日程提醒、文件整理追踪
- 任务能跨天恢复执行

### Week 3: 主动触达与信任层最小版

新增内容：
- 主动卡片/通知
- 高风险动作确认框
- 执行记录时间线
- 失败原因与重试按钮

验收标准：
- 用户能看到“为什么提醒我”
- 用户能看到“你替我做过什么”
- 高风险动作不会静默执行
